  2. Resolve current PID/HWND and process-start truth.
  3. Archive stale proof pointers under docs/recovery/historical/.
  4. Capture fresh RRAPICOORD/API/runtime coordinates.
  5. Run grouped current-PID family snapshots over prior winning/nearby families.
  6. Decode full family groups and rank by displacement tracking.
  7. Require baseline and displaced API references to match separate candidate pose values.
  8. Run current-PID readback and coordinate proof route.
  9. Promote only same-PID, multi-pose, sufficiently displaced, stable candidates.
  10. Run ProofOnly before movement.
  11. Use a short bounded movement smoke before longer navigation.
  12. Commit only a coherent, validated recovery slice.

Improve next time

  • Automate this HTML generation from JSON artifacts once the layout is accepted.
  • Continue owner/module/RVA provenance research from the proven family.
  • Keep grouped snapshot batches and command envelopes; they explain why a candidate won.
  • Add stronger mocked promotion-wrapper execution tests.

  • Keep route readiness separate from movement permission.
  • Keep ProofOnly and MovementSent distinct in all summaries.
