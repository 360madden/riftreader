## Detailed findings

### 1) Watcher script invokes the reader with an invalid mode and its `-Json` path is not machine-clean

- **Severity:** High
- **Area:** scripts / automation
- **Files:**
  - `scripts/watch-readerbridge-export.ps1`
- **Why this is a bug:**
  - The script builds a reader invocation with only `--readerbridge-snapshot-file`, but that flag is now just a file override, not a standalone operation.
  - Its `-Json` path also writes host text, so even a successful run would not be pure JSON.
- **Evidence / repro:**
  - The script builds `@('--readerbridge-snapshot-file', $snapshotPath)`.
  - Direct repro of that shape exits with usage instead of JSON:
    - `dotnet run --project .\reader\RiftReader.Reader\RiftReader.Reader.csproj -- --readerbridge-snapshot-file "<path>" --json`

### 2) Fresh owner-component regeneration is currently broken live by coord-trace timeout

- **Severity:** High
- **Area:** live reverse-engineering workflow
- **Files:**
  - `scripts/capture-player-owner-components.ps1`
  - `scripts/trace-player-coord-write.ps1`
- **Why this is a bug:**
  - The live refresh path needed to regenerate `scripts/captures/player-owner-components.json` currently arms the CE trace, posts movement, and times out without a verified hit.
  - That blocks the fresh owner-component artifact that the stat-hub workflow depends on.
- **Evidence / repro:**
  - Live repro observed:
    - `.\scripts\capture-player-owner-components.ps1 -RefreshSelectorTrace -Json`
  - Failure:
    - `Timed out waiting for a verified coord write trace hit across 1 candidates. 0x2C119477B50:armed`

### 3) `run-reader.cmd` docs show a broken invocation shape

- **Severity:** High
- **Area:** docs / operator workflow
- **Files:**
  - `README.md`
  - `CLAUDE.md`
  - `scripts/run-reader.cmd`
- **Why this is a bug:**
  - `scripts/run-reader.cmd` already inserts the `dotnet run -- ...` separator.
  - Docs that show `run-reader.cmd -- --process-name ...` cause an extra bare `--` to reach the app and break execution.
- **Evidence / repro:**
  - Working:
    - `cmd /c .\scripts\run-reader.cmd --addon-snapshot --json`
  - Broken:
    - `cmd /c .\scripts\run-reader.cmd -- --addon-snapshot --json`

### 4) Refresh flags do not propagate fully through the selector/source-chain/cluster refresh stack

- **Severity:** Medium
- **Area:** scripts / stale artifact handling
- **Files:**
  - `scripts/capture-player-owner-graph.ps1`
  - `scripts/trace-player-selector-owner.ps1`
  - `scripts/capture-player-source-accessor-family.ps1`
  - `scripts/capture-player-source-chain.ps1`
  - `scripts/capture-player-trace-cluster.ps1`
- **Why this is a bug:**
  - Several refresh entry points rerun only one layer of the pipeline and silently reuse stale upstream artifacts.
  - That makes operators think they performed a live refresh when they did not.
- **Evidence / repro:**
  - `capture-player-owner-graph.ps1` reruns selector trace but does not propagate `-RefreshSourceChain`.
  - `capture-player-source-accessor-family.ps1` reruns source-chain capture without `-RefreshCluster`.
  - `capture-player-source-chain.ps1` reruns cluster capture without `-RefreshTrace`.
  - These gaps match the stale-artifact live failures observed after reload.

### 5) Downstream stat workflows claim to refresh owner components but usually reuse stale artifacts

- **Severity:** Medium
- **Area:** scripts / stat workflow correctness
- **Files:**
  - `scripts/capture-player-stat-hub-graph.ps1`
  - `scripts/narrow-player-stat-components.ps1`
  - `scripts/capture-player-owner-components.ps1`
- **Why this is a bug:**
  - The downstream scripts call owner-component capture without `-RefreshSelectorTrace`, so the “refresh” often just reuses stale selector/owner artifacts.
  - This can produce empty or misleading stat-hub results while appearing to succeed.
- **Evidence / repro:**
  - Live session evidence:
    - Snapshot refreshed at `2026-04-09T18:06:43Z`
    - `scripts/captures/player-owner-components.json` still dated `2026-04-09T06:20:31Z`
  - Live result:
    - `--rank-stat-hubs --json` succeeded but returned empty `IdentityComponents` and `RankedSharedHubs`

### 6) Native no-focus post helpers hard-fail on one specific Cheat Engine process name

- **Severity:** Medium
- **Area:** scripts / environment assumptions
- **Files:**
  - `scripts/post-rift-command.ps1`
  - `scripts/post-rift-thread-command.ps1`
- **Why this is a bug:**
  - Both helpers assume the background focus target is exactly `cheatengine-x86_64-SSE4-AVX2`.
  - If the installed CE build uses another process name, the native no-focus path fails before it can verify or perform the post.
- **Evidence / repro:**
  - Live refresh had to fall back because the expected CE process name was unavailable in the current environment.

### 7) Post/reload helpers are pinned to one character's SavedVariables path

- **Severity:** Medium
- **Area:** scripts / environment assumptions
- **Files:**
  - `scripts/post-rift-command.ps1`
  - `scripts/post-rift-thread-command.ps1`
  - `scripts/post-rift-command-ahk.ps1`
- **Why this is a bug:**
  - These scripts verify success by watching a single hardcoded SavedVariables file path under one account/shard/character.
  - On another account or character they can false-timeout even if the in-game command succeeded.
- **Evidence / repro:**
  - All three contain the same literal path under:
    - `C:\Users\mrkoo\OneDrive\Documents\RIFT\Interface\Saved\rift315.1@gmail.com\Deepwood\Atank\SavedVariables\ReaderBridgeExport.lua`

### 8) Stat-hub JSON output loses the generated probe-script path

- **Severity:** Medium
- **Area:** reader / JSON contract
- **Files:**
  - `reader/RiftReader.Reader/Program.cs`
  - `reader/RiftReader.Reader/Models/PlayerStatHubRankResult.cs`
- **Why this is a bug:**
  - `--rank-stat-hubs --cheatengine-stat-hubs --json` generates a Lua file as a side effect, but the JSON response does not include the output path.
  - Automation can’t discover where the generated script was written.
- **Evidence / repro:**
  - `RunStatHubRankingMode` writes the script, but serializes only `PlayerStatHubRankResult`

### 9) `--readerbridge-snapshot-file` support is only partially documented

- **Severity:** Medium
- **Area:** reader / CLI discoverability
- **Files:**
  - `reader/RiftReader.Reader/Cli/ReaderOptionsParser.cs`
  - `reader/RiftReader.Reader/Program.cs`
- **Why this is a bug:**
  - Several modes consume `ReaderBridgeSnapshotFile`, but help text documents it only for snapshot mode and stat-hub ranking.
  - Supported behavior is therefore easy to miss and appears inconsistent.
- **Evidence / repro:**
  - `Program.cs` uses the override in multiple modes including cheatengine-probe, capture-readerbridge-best-family, read-player-current, read-player-coord-anchor, and rank-stat-hubs

### 10) `ReaderBridgeExport` README documents the wrong fallback behavior

- **Severity:** Medium
- **Area:** addon docs
- **Files:**
  - `addon/ReaderBridgeExport/README.md`
  - `addon/ReaderBridgeExport/main.lua`
- **Why this is a bug:**
  - The README says the addon stays in a waiting state if `ReaderBridge` is missing.
  - The code actually falls back to a direct Rift API snapshot and can report `status = "ready"` with `sourceMode = "DirectAPI"`.
- **Evidence / repro:**
  - Code path in `main.lua` builds a direct snapshot fallback instead of waiting-only behavior

### 11) Validation spec promises startup capture that the addon does not perform

- **Severity:** Medium
- **Area:** docs / actual behavior mismatch
- **Files:**
  - `docs/addon-validation-spec.md`
  - `addon/RiftReaderValidator/main.lua`
- **Why this is a bug:**
  - The spec lists startup as an automatic capture trigger.
  - The code does not capture on startup; first automatic capture happens later via the update loop as `auto.available`.
- **Evidence / repro:**
  - `OnStartup()` initializes state/UI but does not call `Validator.Capture(...)`

### 12) Reverse-engineering docs contain stale or incomplete operational guidance

- **Severity:** Medium
- **Area:** tools / docs
- **Files:**
  - `tools/reverse-engineering/README.md`
  - `tools/reverse-engineering/install-tools.ps1`

- **Why this is a bug:**

### 13) `--rank-stat-hubs` help text is slightly misleading about process targeting

### 14) Stat-hub-only Cheat Engine scripts still expose family-oriented UX when no families exist

## Current most important blocker

The biggest functional blocker right now is the live refresh pipeline used to regenerate owner-component artifacts:

- ReaderBridge snapshot refresh is working
- Live stat-hub ranking runs
- Fresh owner-component regeneration is not completing
- Downstream stat-hub workflows therefore consume stale owner-component artifacts and can return empty results

If fixing starts later, this should be the first area to prioritize.
